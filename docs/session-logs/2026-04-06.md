#### v0.18.6 — Close concurrent-submission and double-confirmation races
- `createSubmission` (match.repository.ts): replaced unconditional match status update with `updateMany WHERE status IN (PENDING, IN_PROGRESS)`. If count=0, throws `MATCH_ALREADY_SUBMITTED` to roll back the transaction including the just-created submission row.
- `confirmSubmission` (match.repository.ts): replaced unconditional submission update with `updateMany WHERE status = PENDING`. If count=0, throws `ALREADY_CONFIRMED` to roll back, preventing double rating application.
- `match.service.ts`: catches both sentinel errors; `applyRatingResult` only runs after `confirmSubmission` succeeds.

#### v0.18.7 — Fix BOTH verification mode (OR logic + UI context)
- Backend: restructured `confirmMatchResult` verification into clean if-else-if branches. BOTH + non-self-confirm now uses OR logic (either code OR birth year sufficient). BOTH + self-confirm remains birth-year-only.
- UI: `confirm/page.tsx` computes `isSelfConfirm` and passes it to `ConfirmResultForm`. Heading updated to say "or" for BOTH non-self-confirm, and birth-year-only copy for BOTH self-confirm. Form hides the code field for BOTH self-confirmers.
- Tests: updated BOTH non-self-confirm test cases from old AND-required to new OR semantics.

#### v0.19.0 — Security Hardening
- **Secrets audit**: clean — no hardcoded secrets, `.env` gitignored, no client-side env var exposure
- **CORS audit**: N/A — no custom API routes; Next.js same-origin default applies
- **IDOR audit**: clean — `isAuthorizedAsTD`, `isMatchParticipantOrTD`, `isPlatformAdmin` chain covers all mutation paths
