# Codex Brief — v0.17.2: Rating Snapshot + Group Draws

## Goal

Preserve each entrant's rating at the time of bracket generation, display it instead of the live rating in the entrants list and the groups section, and rename the groups section "Group Draws" with fixed draw-order row sorting.

## Service Changes — `src/server/services/bracket.service.ts`

### 1. `generateRoundRobinBracket` (line ~296)

Currently builds a `ratings` parallel array and a `playerIds` array, then calls `assignGroups`. After that, it stamps `groupNumber` with `updateMany` (all players in the group share the same value).

Since `ratingSnapshot` differs per player, replace the `updateMany` in the transaction loop with per-player `update` calls. Build a lookup map before the loop.

**Change:**

```typescript
// Before the $transaction:
const playerRatingMap = new Map(playerIds.map((id, i) => [id, ratings[i]]));

// Inside the $transaction, replace:
await tx.eventEntry.updateMany({
  where: { eventId, playerProfileId: { in: groupPlayerIds } },
  data: { groupNumber },
});

// With:
for (const pid of groupPlayerIds) {
  await tx.eventEntry.update({
    where: { eventId_playerProfileId: { eventId, playerProfileId: pid } },
    data: { groupNumber, ratingSnapshot: playerRatingMap.get(pid) ?? null },
  });
}
```

The `ratings[i]` values are already sourced from `e.playerProfile.playerRatings.find(r => r.ratingCategoryId === ratingCategoryId)?.rating ?? 1500`. This is the correct snapshot value — use it as-is.

### 2. `generateBracket` — SE branch (line ~291)

For `SINGLE_ELIMINATION` events, `generateBracket` calls `generateSingleEliminationBracket(eventId, event.eventEntries)`. At this point `event` is fully loaded (it includes `playerProfile.playerRatings`). Before calling the generator, stamp `ratingSnapshot` on all EventEntries in a transaction:

```typescript
// Add before the existing `await generateSingleEliminationBracket(...)` call:
const { ratingCategoryId } = event;
await prisma.$transaction(
  event.eventEntries.map((e) => {
    const snap =
      e.playerProfile.playerRatings.find((r) => r.ratingCategoryId === ratingCategoryId)
        ?.rating ?? null;
    return prisma.eventEntry.update({
      where: { eventId_playerProfileId: { eventId, playerProfileId: e.playerProfileId } },
      data: { ratingSnapshot: snap },
    });
  }),
);
```

`RR_TO_SE` events go through `generateRoundRobinBracket` (the RR fix above covers them at group formation time).

## UI Changes

### A. Event detail page — `src/app/tournaments/[id]/events/[eventId]/page.tsx`

#### 1. Entrants list (lines ~358–392)

Currently sorts by live `playerRatings.rating` and displays it.

**New behavior:** 
- For display: use `entry.ratingSnapshot` if not null, else fall back to `playerRatings.rating` (for pre-start events with no snapshot yet).
- For sort: use `ratingSnapshot` as the sort key with the same fallback.

Change the sort comparator and the displayed rating value accordingly. The fallback logic should be extracted into a small helper at the top of the computation (inside the page function):

```typescript
function getDisplayRating(entry: typeof event.eventEntries[number]): number | null {
  if (entry.ratingSnapshot != null) return entry.ratingSnapshot;
  return entry.playerProfile.playerRatings.find(
    (r) => r.ratingCategoryId === event.ratingCategoryId,
  )?.rating ?? null;
}
```

Use `getDisplayRating(entry)` everywhere you previously computed the rating inline for the entrants list.

#### 2. Groups section (lines ~65–148 for data assembly, ~410–476 for render)

**Data assembly changes (lines ~109–146):**

The `rawPlayers` array currently reads `e.playerProfile.playerRatings.find(...)?.rating` for each player. Change this to use `e.ratingSnapshot ?? ratingRow?.rating ?? null` (same fallback pattern as entrants list).

The **row sort order** must now always be by `ratingSnapshot DESC` (draw order), regardless of `groupComplete`. Remove the conditional `[...rawPlayers].sort((a, b) => b.wins - a.wins)` sort. Instead:

```typescript
// Always sort by ratingSnapshot desc (draw order)
const drawOrderPlayers = [...rawPlayers].sort((a, b) => {
  const aRating = a.ratingSnapshot ?? a.rating ?? 0;
  const bRating = b.ratingSnapshot ?? b.rating ?? 0;
  return bRating - aRating;
});
```

The **rank assignment** currently relies on the wins-sorted order. Since rows no longer re-sort by wins, compute ranks separately from a wins-sorted copy, then look them up by `playerProfileId`:

```typescript
const rankMap = new Map<string, number>();
if (groupComplete) {
  const winsSorted = [...rawPlayers].sort((a, b) => b.wins - a.wins);
  for (let i = 0; i < winsSorted.length; i++) {
    const prev = winsSorted[i - 1];
    const rank =
      i === 0 ? 1
      : winsSorted[i].wins === prev.wins ? rankMap.get(prev.playerProfileId)!
      : i + 1;
    rankMap.set(winsSorted[i].playerProfileId, rank);
  }
}

groups.push({
  groupNumber: groupNum,
  groupComplete,
  players: drawOrderPlayers.map((p) => ({
    ...p,
    rank: groupComplete ? (rankMap.get(p.playerProfileId) ?? null) : null,
  })),
});
```

Note: add `ratingSnapshot: number | null` to the `players` type in the `groups` array type definition at line ~68.

**Render changes:**

- Line ~412: Change `"Groups"` heading text to `"Group Draws"`.
- The rank column (`#`) and rank cell already render based on `p.rank` — no change needed there since `rank` is still passed per-player. The rows are just in different order now.
- The Rtg column: it currently displays `p.rating`. This is already correct since we updated `rawPlayers` to use the snapshot with fallback.
